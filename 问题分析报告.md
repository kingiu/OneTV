# "危险关系"剧集视频源重复问题分析报告

## 问题描述

用户通过热门剧集界面点击打开"危险关系"剧集后，视频源列表中出现全部视频源重复显示的问题。

## 问题复现

### 测试场景
1. 从热门剧集页面导航到"危险关系"剧集详情页
2. 点击播放按钮进入播放页面
3. 打开播放源选择器（SourceSelectionModal）
4. 观察视频源列表

### 测试结果
根据测试脚本 `test-dangerous-liaisons-debug.ts` 的输出：
- 原始搜索结果：3 个结果
- 去重后：2 个结果（存在重复数据）
- 重复项：索引 1 与索引 0 重复（jszyapi.com_123_危险关系_2024）

## 问题根本原因分析

### 1. 数据层问题（detailStore.ts）

#### 问题位置：`processAndSetResults` 函数（第 100-290 行）

**问题描述**：去重逻辑存在缺陷，导致相同 source + id + title + year 的结果被错误地保留多个。

**关键代码**（第 130-160 行）：
```typescript
// 合并模式（merge=true）下的去重逻辑
if (merge) {
  const seenIds = new Set<string>();
  const seenSourceTitleYear = new Set<string>();
  const existingSearchResults = state.searchResults || [];
  
  // 先添加已有的搜索结果
  finalResults = existingSearchResults.filter((r) => {
    const uniqueKey = `${r.source}_${r.id}`;
    const sourceTitleYearKey = `${r.source}_${r.title}_${r.year || ''}`;
    
    if (seenIds.has(uniqueKey)) {
      return false;  // 问题：这里过滤掉重复的 uniqueKey
    }
    
    if (!seenSourceTitleYear.has(sourceTitleYearKey)) {
      seenSourceTitleYear.add(sourceTitleYearKey);
      seenIds.add(uniqueKey);
      return true;
    }
    
    return false;
  });
```

**问题分析**：
- 去重使用了 `uniqueKey = source_id` 和 `sourceTitleYearKey = source_title_year` 两个键
- 但两个键的检查逻辑不一致，导致某些情况下重复项未被正确过滤
- 特别是在 `merge=true` 模式下（后台搜索），已有的结果和新的结果分别去重后再合并，可能导致重复

#### 问题位置：`episodesSelectorBySource` 选择器（第 713-714 行）

```typescript
export const episodesSelectorBySource = (source: string) => (state: DetailState) =>
  state.searchResults.find((r) => r.source === source)?.episodes || [];
```

**问题描述**：该选择器只返回第一个匹配 source 的结果，但如果 searchResults 中存在多个相同 source 的重复项，只会返回第一个，导致剧集数据不完整。

### 2. 播放层问题（playerStore.ts）

#### 问题位置：`loadVideo` 函数（第 215-224 行）

```typescript
let episodes: string[] = [];

// If detail exists, use detail.source; otherwise use provided source.
if (detail && detail.source) {
  logger.info(`[INFO] Using existing detail source "${detail.source}" to get episodes`);
  episodes = episodesSelectorBySource(detail.source)(useDetailStore.getState());
} else {
  logger.info(`[INFO] No existing detail, using provided source "${source}" to get episodes`);
  episodes = episodesSelectorBySource(source)(useDetailStore.getState());
}
```

**问题描述**：
- 当 `detail.source` 在 searchResults 中有多个重复项时，`episodesSelectorBySource` 只返回第一个
- 但后续的 `selectBestSource` 会重新选择最佳源，可能导致切换到其他重复的源

#### 问题位置：`mapEpisodesForPlayback` 函数（第 13-33 行）

```typescript
const mapEpisodesForPlayback = (episodeUrls: string[], sourceKey: string) => {
  const { apiBaseUrl, vodAdBlockEnabled } = useSettingsStore.getState();

  return episodeUrls.map((episodeUrl, index) => {
    const playbackCandidates = getPlaybackUrlCandidates(episodeUrl, apiBaseUrl, sourceKey, vodAdBlockEnabled);

    const uniqueCandidates = Array.from(
      new Set(
        [...(playbackCandidates.length > 0 ? playbackCandidates : []), episodeUrl].filter(
          (candidate): candidate is string => Boolean(candidate)
        )
      )
    );

    return {
      url: uniqueCandidates[0] || episodeUrl,
      rawUrl: episodeUrl,
      title: getEpisodeTitle(index),
      urlCandidates: uniqueCandidates,
      currentCandidateIndex: 0,
    };
  });
};
```

**问题描述**：该函数本身没有问题，它会对单个剧集的播放候选 URL 进行去重，但无法解决上游数据重复的问题。

### 3. 渲染层问题（EpisodeSelectionModal.tsx）

#### 问题位置：线路选择渲染（第 97-110 行）

```typescript
{playSources.length > 0 ? (
  playSources.map((source: any, index) => (
    <StyledButton
      key={index}
      text={source.name || `线路${index + 1}`}
      onPress={() => onLineChange(index)}
      isSelected={currentPlaySourceIndex === index}
      style={styles.lineItem}
      textStyle={styles.lineItemText}
    />
  ))
) : (
  <Text style={{ color: 'white', textAlign: 'center', marginTop: 20 }}>暂无可用线路</Text>
)}
```

**问题描述**：
- `playSources` 来自 `detail?.play_sources`（第 14 行）
- 如果 `detail` 对象的 `play_sources` 本身包含重复的线路数据，渲染层会正常渲染所有线路
- 问题根源在于 `detail` 对象的 `play_sources` 数据重复

### 4. 数据流问题

```
API 搜索 → detailStore.init → processAndSetResults (去重失败) → 
searchResults 包含重复项 → detail 选择第一个 → 
detail.play_sources 可能包含重复线路 → 
EpisodeSelectionModal 渲染所有线路（包括重复的）
```

## 问题定位

### 高风险代码模块

1. **detailStore.ts** - 第 100-290 行的 `processAndSetResults` 函数
   - 去重逻辑复杂且存在缺陷
   - merge 模式和非 merge 模式的去重逻辑不一致

2. **detailStore.ts** - 第 713-714 行的 `episodesSelectorBySource` 选择器
   - 只返回第一个匹配项，无法处理重复数据

3. **detailStore.ts** - 第 616-708 行的 `selectBestSource` 函数
   - 在重复的源中选择最佳源，可能导致不必要的源切换

### 问题触发条件

1. 后端 API 返回重复的搜索结果（相同 source + id + title + year）
2. detailStore 的去重逻辑未能正确过滤重复项
3. 重复的源被添加到 searchResults
4. detail 选择其中一个作为当前源
5. EpisodeSelectionModal 渲染该源的所有线路

## 影响范围

- **前端渲染**：EpisodeSelectionModal 的线路选择列表
- **播放体验**：用户可能看到重复的线路选项
- **性能影响**：额外的数据处理和内存占用

## 修复建议

### 优先级 P0 - 立即修复

#### 1. 修复 detailStore.ts 的去重逻辑

**问题**：`processAndSetResults` 函数的去重逻辑存在缺陷

**修复方案**：
- 统一去重逻辑，使用 `source_id_title_year` 作为唯一键
- 移除复杂的 merge/non-merge 分支逻辑
- 确保所有场景下都正确去重

**代码位置**：detailStore.ts 第 100-290 行

#### 2. 增强 episodesSelectorBySource 的健壮性

**问题**：当存在重复源时，只返回第一个

**修复方案**：
- 添加日志记录重复源的情况
- 考虑返回所有匹配源的合并数据（如果合理）

**代码位置**：detailStore.ts 第 713-714 行

### 优先级 P1 - 建议修复

#### 3. 添加数据验证中间件

**建议**：在 API 响应后、store 更新前添加数据验证层
- 检测重复数据
- 记录重复数据的来源
- 可选：自动合并重复数据

#### 4. 添加单元测试

**建议**：为 `processAndSetResults` 函数添加单元测试
- 测试重复数据场景
- 测试 merge/non-merge 场景
- 确保去重逻辑正确

## 总结

**核心问题**：detailStore.ts 中的 `processAndSetResults` 函数去重逻辑存在缺陷，导致相同 source + id + title + year 的搜索结果未被正确过滤，最终在 EpisodeSelectionModal 中渲染重复的视频源。

**修复重点**：
1. 修复去重逻辑，确保使用统一的唯一键
2. 简化 merge/non-merge 分支逻辑
3. 添加单元测试验证

**影响评估**：
- 中等影响：主要影响用户体验（重复线路显示），不影响核心播放功能
- 修复优先级：P0（高）
